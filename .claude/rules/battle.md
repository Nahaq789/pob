# Battle Design

## Redis State

| 種別 | 内容 |
|------|------|
| 固定 | ベース実数値（nature補正済み）、タイプ、技リスト |
| 動的 | 現在HP、能力ランク（−6〜+6）、PP、状態異常、道具、特性 |

## Damage Calculation

- Next.js → battle-service へのリクエストは**技IDのみ**
- battle-service が Redis から状態取得して計算を完結させる
- 乱数: 0.85〜1.00 の 16段階からランダム選択
- タイプ相性テーブルは battle-service に**静的データ**として保持

## Validation（battle-service）

1. 技IDがポケモンの技リストに存在するか
2. PPが残っているか
3. 自分のターンか

## PP Management

- アイテムによる回復なし
- PP消費は battle-service が計算しレスポンスで返す（通常: −1、Pressure特性: −2）
- PP = 0 の技はフロントで非活性化

## Move/Ability Handler Pipeline

介入タイミング（5種）:
1. 技使用前
2. ダメージ計算前
3. ダメージ計算中
4. ダメージ計算後
5. ターン終了時

各タイミングにハンドラー配列を用意し、優先度でソート（タイプ変更系を先に処理）。

| Tier | 種別 |
|------|------|
| A | 汎用ハンドラー（多くの技・特性に共通） |
| B | 汎用ハンドラー（複数共有できる特殊処理） |
| C | 個別ハンドラー（固有処理） |

レジストリ方式で管理。

## Notes

- 全 900+ 技の実装は不要
- 拡張可能なアーキテクチャ + README での説明（「X技実装済み、構造は容易に拡張可能」）が重要
